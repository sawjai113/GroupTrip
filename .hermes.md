# Wanderaid / Group Trip App

Hermes project rules for the Wanderaid SwiftUI iOS app.

## Build & Test

```sh
xcodebuild -project "GroupTripApp.xcodeproj" -scheme GroupTripApp -destination "generic/platform=iOS" CODE_SIGNING_ALLOWED=NO build
xcodebuild test -project "GroupTripApp.xcodeproj" -scheme GroupTripApp -destination 'platform=iOS Simulator,name=iPhone 17' CODE_SIGNING_ALLOWED=NO
```

## Development Rules

- Test-driven development for behavior changes, bug fixes, and refactors.
- Preserve user changes. Do not revert unrelated work.
- Prefer small, reviewable changes over broad rewrites.
- Keep Supabase keys client-safe only. Never add service-role keys to the app.
- Any action that deletes, removes, revokes, logs out, leaves, or otherwise destroys access must include a confirmation step before it completes.
- Avoid growing any single SwiftUI file into a catch-all surface.
- Run `git diff --check` before commit (no trailing whitespace).
- Secret-scan modified files before commit.
- Do not push commits unless explicitly asked.

## UX Conventions

- App name: Wanderaid
- Google OAuth primary auth; email/password backup
- Destructive actions must have confirmation
- todo/feedback tracked in docs/todo-feedback.md

## Subagent Delegation Strategy

Use subagents for work that benefits from isolation, skip them for work that doesn't. These rules guide the decision:

### Delegate to a subagent when:
- Multi-file pattern with established skill (wanderaid-cloud-persistence-pattern, wanderaid-tdd-sync-boilerplate)
- Cross-cutting work that spans layers (schema + service + store + tests)
- Research / design proposal (read several files, synthesize a plan)
- Any implementation task that touches 4+ files or involves non-trivial logic
- QA review pass after any non-trivial change

### Do NOT delegate (work inline) when:
- Single-file mechanical change (rename, typo fix, add a single button)
- File I just read and can patch directly
- Trivial one-line fixes
- Exploration / spike where I need to iterate fast

### QA gate:
- Any non-trivial change (delegated or inline) gets a QA review pass before commit.
- For inline work touching 2+ files or containing logic changes: spawn QA/Release & PR Review Agent to review the diff before committing.
- For delegated work: always run QA/Release & PR Review Agent after the implementation agent finishes.
- QA review skips on: typo-only fixes, docs-only changes, whitespace fixes, single-line trivial fixes.

### Delegation tips:
- Always load the relevant wanderaid-* skill and pass its context before delegating.
- Pass the full current test count and build status in the subagent context so it knows the baseline.
- For batch work (e.g. add 3 new entity types), use delegate_task(tasks=[]) to run in parallel.
- After delegation finishes, verify the result with a targeted test run — subagent summaries are self-reported.
- Never delegate: clarify, memory, send_message, cron creation, or anything requiring user interaction.

## Pre-Commit Checklist

Before marking any non-trivial task complete (skip for single-line/typo/docs fixes):
0. If inline work touching 2+ files or logic changes: run QA review subagent first.
1. If delegated: confirm QA/Release & PR Review Agent passed.
2. Run the full test suite (xcodebuild test ...)
3. Run the generic build (xcodebuild build ...)
4. Run `git diff --check`
5. Run secret scan on modified files
6. If a subagent did the work, run the targeted test for the changed area and verify it passes
